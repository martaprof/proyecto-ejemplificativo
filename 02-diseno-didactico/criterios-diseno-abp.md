# Criterios para diseñar el proyecto como ABP

Este documento recoge los elementos que debe incluir el proyecto ejemplificativo para ajustarse a una metodología de Aprendizaje Basado en Proyectos (ABP).

## 1. Sentido del proyecto

El proyecto debe cumplir dos condiciones generales:

- tener sentido para el alumnado, conectando con su realidad, intereses o problemas cercanos;
- tener un objetivo claramente definido, significativo y relacionado con el currículo.

No debe plantearse como una simple práctica técnica, sino como una experiencia en la que el alumnado necesite aprender contenidos de Sistemas Electrónicos para resolver un reto.

## 2. Pregunta guía, problema o reto

El proyecto debe partir de una pregunta guía, problema o reto inicial.

La pregunta guía debe ser:

- abierta, sin una única respuesta cerrada;
- motivadora y comprensible para el alumnado;
- conectada con una situación real o verosímil;
- alcanzable para 1.º de Bachillerato;
- vinculada a los contenidos curriculares;
- orientada a desarrollar pensamiento crítico, síntesis, aplicación y toma de decisiones.

En este proyecto, la pregunta guía será:

> ¿Cómo podemos diseñar un sistema electrónico con Arduino que ayude a conocer las condiciones atmosféricas de un invernadero y facilite la toma de decisiones para su cuidado?

## 3. Producto final tangible

El ABP debe conducir a un producto final concreto. En este proyecto podrá ser:

- un prototipo simulado en Tinkercad;
- un montaje físico con Arduino;
- una memoria técnica;
- una presentación pública;
- un vídeo o demostración del funcionamiento;
- una guía de uso del sistema diseñado.

El producto final debe responder a la pregunta guía o reto inicial.

## 4. Contenido significativo

El proyecto debe seleccionar contenidos relevantes de la materia y trabajarlos en profundidad.

En el área de Sistemas Electrónicos, se incluyen contenidos como:

- sensores y actuadores;
- entradas y salidas digitales;
- entradas analógicas;
- programación básica con Arduino;
- montaje y conexión de circuitos;
- lectura de esquemas;
- depuración de errores;
- diseño de soluciones técnicas;
- uso de simuladores como Tinkercad.

Los contenidos no deben aparecer aislados: deben necesitarse para avanzar en el reto.

## 5. Necesidad de saber

El proyecto debe comenzar con una actividad motivadora que genere curiosidad, preguntas o una necesidad real de aprender.

Posibles actividades iniciales:

- presentación de una situación problemática cercana;
- análisis de un objeto o sistema electrónico cotidiano;
- demostración de un prototipo incompleto;
- debate sobre automatización, sensores o tecnología en la vida diaria;
- pequeño reto diagnóstico con Arduino o Tinkercad.

La finalidad es que el alumnado perciba que necesita adquirir nuevos conocimientos para resolver el proyecto.

## 6. Autonomía del alumnado

El alumnado debe tener margen real de decisión durante el proyecto.

La autonomía puede aparecer en:

- elección del problema concreto dentro de un marco común;
- diseño de la solución;
- reparto de roles en el equipo;
- búsqueda de información;
- selección de componentes;
- decisiones de montaje;
- forma de presentar el producto final;
- revisión y mejora del prototipo.

El docente actúa como guía, orientador y dinamizador, no como única fuente de información.

## 7. Plan de trabajo

El proyecto debe incluir una planificación clara de tareas, fases, recursos y tiempos.

Como mínimo, debe distinguir tres fases:

1. Investigación y planificación.
2. Desarrollo del producto final.
3. Exposición, evaluación y revisión.

Cada fase debe indicar tareas del alumnado, papel del docente, recursos necesarios, evidencias generadas, tiempo estimado y criterios de seguimiento.

## 8. Investigación e innovación

El alumnado debe realizar un proceso de indagación, no limitarse a seguir instrucciones cerradas.

El proyecto debe incluir momentos para formular preguntas, buscar información, comparar alternativas, probar soluciones, detectar errores, revisar el diseño, justificar decisiones y extraer conclusiones.

El proceso debe ser cíclico: investigar, probar, revisar y mejorar.

## 9. Trabajo individual y cooperativo

El ABP debe combinar responsabilidad individual y trabajo en equipo.

El diseño debe contemplar equipos de trabajo, roles o responsabilidades, momentos de decisión compartida, registro de avances, comunicación entre integrantes y coevaluación del trabajo en equipo.

## 10. Evaluación, retroalimentación y revisión

La evaluación debe estar presente durante todo el proceso, no solo al final.

El proyecto debe incluir:

- rúbrica desde el inicio;
- seguimiento del proceso;
- retroalimentación del docente;
- revisión entre iguales;
- autoevaluación;
- mejora del producto tras recibir comentarios;
- evaluación del producto final y del proceso seguido.

La rúbrica debe definir varios niveles de logro para los contenidos y competencias trabajadas.

## 11. Presentación pública o difusión

El producto final debe presentarse ante una audiencia: grupo clase, otros docentes, alumnado de otros grupos, familias, comunidad educativa o repositorio público en GitHub.

La presentación debe mostrar el problema abordado, la solución propuesta, el funcionamiento del prototipo, las decisiones técnicas, las dificultades encontradas y las mejoras posibles.

## 12. Competencias y habilidades a desarrollar

El proyecto debe contribuir al desarrollo de:

- pensamiento crítico;
- resolución de problemas;
- planificación;
- colaboración;
- comunicación;
- toma de decisiones;
- gestión del tiempo;
- uso de TIC;
- aprender a aprender;
- aplicación de conocimientos al mundo real.

## 13. Condiciones de aplicación

Para que el proyecto sea viable, deben quedar definidos:

- duración aproximada;
- número de sesiones;
- recursos disponibles;
- conocimientos previos necesarios;
- nivel de guía docente;
- herramientas técnicas;
- organización de equipos;
- espacios necesarios;
- posibles dificultades;
- adaptaciones si no se dispone de material físico.

## 14. Evidencias que debería recoger el repositorio

El repositorio del proyecto debería incluir:

- README general del proyecto;
- guía docente;
- pregunta guía y reto inicial;
- temporalización;
- desarrollo por sesiones;
- materiales para el alumnado;
- orientaciones para el docente;
- recursos de Arduino IDE y Tinkercad;
- rúbricas;
- autoevaluación y coevaluación;
- ejemplos de producto final;
- instrucciones de presentación pública;
- propuestas de mejora y reutilización.

## 15. Lista de comprobación ABP

Antes de cerrar el diseño, el proyecto debería responder afirmativamente a estas preguntas:

- ¿Existe una pregunta guía, problema o reto claro?
- ¿El reto conecta con la realidad o intereses del alumnado?
- ¿El proyecto trabaja contenidos curriculares relevantes?
- ¿El alumnado necesita investigar y tomar decisiones?
- ¿Hay un producto final tangible?
- ¿El proceso está dividido en fases y plazos?
- ¿Hay autonomía real del alumnado?
- ¿Se trabaja de forma cooperativa?
- ¿Se proporcionan rúbricas desde el inicio?
- ¿Hay retroalimentación y revisión durante el proceso?
- ¿Se evalúa tanto el proceso como el producto final?
- ¿Hay autoevaluación y evaluación entre iguales?
- ¿El producto se presenta o difunde públicamente?
- ¿El repositorio permite que otro docente entienda y reutilice el proyecto?
